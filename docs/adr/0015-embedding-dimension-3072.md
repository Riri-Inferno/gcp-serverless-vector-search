# ADR 0015: 埋め込みベクトルの次元数を 1536 → 3072 に変更

- **Date**: 2026-06-23
- **Status**: Accepted
- **Supersedes**: [0001](0001-api-design-mvp.md)（Consequences の `vector(1536)`）, [0004](0004-firestore-embedding-model-field.md)（スキーマの `vector(1536)`）, [0005](0005-cloud-functions-runtime-stack.md)（`output_dimensionality: 1536`）

## Context

[ADR 0001](0001-api-design-mvp.md) で埋め込み次元を `1536` に決定した。当時の根拠は「推奨値 768/1536/3072 の中間値。精度とストレージのバランスが良い」という判断だったが、具体的な検証はなかった。

その後、マルチモーダル対応（[PR: feature/terraform-add-gcs-multi-media-20260623](../../)）を追加するにあたり、画像・音声・PDF などのクロスモーダル検索（テキストクエリで画像がヒットする）の品質を確認した。ここで以下の事実が判明した：

1. **`output_dimensionality` は切り詰め処理**: `gemini-embedding-2` は常に 3072 次元を生成し、`output_dimensionality` 指定で末尾を切り捨てる仕組み。1536 を指定してもモデル側の計算量は変わらない。つまり **1536 を使っていた時点で 3072 分の計算コストを払っていた**。
2. **クロスモーダル検索は次元が多いほど有利**: テキスト・画像・音声を同一ベクトル空間に射影するマルチモーダル埋め込みでは、異なるモダリティ間の距離を正確に表現するために高次元が必要。

3. **デメリットが実質ない**: 個人ラボ規模では、次元増加によるストレージ増（~6KB/doc）・検索レイテンシ増（O(n×3072) vs O(n×1536)）・ネットワーク転送量増はいずれも誤差レベル。

また、マルチモーダル対応を加えるこのタイミングが「Firestore コレクションを空にして再インデックス」が最も安いタイミングであった（データがまだ少ない）。

## Decision

`output_dimensionality` を `1536` → `3072` に変更し、Firestore ベクトルインデックスの `dimension` も `3072` に揃える。

**変更箇所:**

| ファイル                       | 変更内容                              |
| ------------------------------ | ------------------------------------- |
| `functions/vector-api/main.py` | `EMBEDDING_DIMENSION = 1536` → `3072` |
| `terraform/gcp/firestore.tf`   | `dimension = 1536` → `3072`           |
| `docs/multimodal-design.md`    | 図・本文の `1536` を `3072` に統一    |
| `README.md`                    | `1536次元` → `3072次元`               |

既存の ADR（0001/0004/0005）の本文は変更しない。当時の決定を歴史的記録として保持し、本 ADR が上書きする。

## Consequences

- Firestore ベクトルインデックスの `dimension` は変更不可（ForceNew）のため、`terraform apply` 前に `documents` コレクションを空にする必要がある
- 既存の 1536 次元ベクトルは 3072 次元インデックスとは互換性がない。コレクションを空にした後、全ドキュメントを再登録（再埋め込み）する
- 検索精度はクロスモーダルを含めて向上する見込み（実測は今後）
- ストレージ: ~6KB/doc の増加（誤差）
- 検索レイテンシ: 個人ラボ規模（数百〜数千件）では体感変化なし
